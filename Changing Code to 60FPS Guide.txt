0) gDeltaTime

The core variable that defines this entire project is gDeltaTime, which is a global variable defined as 30 divided by the framerate (so 30/30, 30/60, 30/120), etc. That results in values of 1 for 30fps, 0.5 for 60fps, and 0.25 for 120FPS. When you want something to go slower, multiply by gDeltaTime (e.g. x+=1 to x+=1*gDeltaTime). When you want something to go faster, divide. Etc.

When doing current_frame stuff, mod by 1/gDeltaTime. So "if current_frame mod 1/gDeltatime == 0" would equate to "every other frame" on 60fps, and "every forth frame" on 120fps, etc. This makes the thing happen at the same rate no matter the FPS - if the game is going 4x as fast, it only happens 1/4th as often. 

1) Common Patterns We Want To Modify

When clicking an object, first look to see if it's a child of a more widely used parent. You may need to modify the parent instead, or modify both. If the object is likely to be created dynamically (e.g. bullets, particles, etc), also do a code search for the object's name to see what scripts/events manually create instances with `instance_create` and then overwrite their properties after their create event runs.

Now that you understand the object, look in the Create event for any variables that stand out. Reason about them, don't touch them yet.

Now look in the Step, Begin Step, End Step, and Draw events, if they exist.

Look for specific value threshold checks, and look for weird arcs and multiplications. If the logic is too complex or spooky for you to understand, consider using `gDeltaDoTicks` (which is 1 on frames that correspond to 30fps frames, and 0 otherwise) to run the object at a lower framerate for now, until it can be rewritten properly later.

If you want to delta-time-ify the object properly, look for the following patterns in code branches that run EVERY FRAME (not only when specific conditions are temporarily met):

-`x += 1` or similar
- - Replace with `x += gDeltaTime`
-`somevar -= someothervar` or similar
- - Replace with `somevar -= someothervar * gDeltaTime`
- If there is a gravity calculation, like `yvel += grav`, change that like `yvel += grav * gDeltaTime` specifically
- moveTo(xVel,yVel) 
- - Replace with moveTo(xVel * gDeltaTime,yVel * gDeltaTime) 


The general pattern is "if this is adding a specific number every frame, multiply the number being added by gDeltaTime".

2) SPEED

Do not change speed/hspeed/vspeed/direction with a simple gDeltaTime modification.

Game Maker applies these variables automatically with its own integration, and fighting it by undoing it and redoing it manually can cause performance problems or weird glitches, when the object in question is not purely cosmetic. Changing the contents to include delta time as a multiplier might seem to work, but if any code in OTHER objects or scripts reads or writes these variables, it will break THAT code. Game Maker is unfortunately just too inflexible to let us deal with this in the correct simple way, so we have to do it the stupid hard way:

1 - Go to the create event and define _direction and _speed (or, if it uses hspeed/vspeed instead, _hspeed and _vspeed).
2 - If there is no direction or speed, define as 0.
3 - Change existing code in the step/draw/etc events to use the _ prefixed versions of the variables instead of the originals.
3.5 - Also look for any instances of the object, or its child objects, being spawned with `instance_create` using the code search tool, and update those locations too, to use `_speed`/`_direction`/etc instead of `speed`/`direction`/etc.
4 - Add the following code to the end of the Step or End Step event (whichever exists):

(if speed/direction)
```c
x += cos(degtorad(_direction)) * _speed * gDeltaTime
y += sin(degtorad(_direction)) * _speed * gDeltaTime
```
(if hspeed/vspeed)
```c
x += _hspeed * gDeltaTime
y += _vspeed * gDeltaTime
```

If both speed/direction AND hspeed+vspeed are used (e.g. speed and hspeed), Brian or Wareya must be summoned to reason about what the object is doing and fix it to only use one of the two systems.

5 - So, for example, something like "speed=bulletSpeed" should be commented out, then replaced with 

`_speed=bulletSpeed`

and then later, exactly once per step event, at or near the end:

```c
x += cos(degtorad(_direction)) * _speed * gDeltaTime
y += sin(degtorad(_direction)) * _speed * gDeltaTime
```

2.2) CHILDREN WITH SPEED

All instances and children of an object must be adjusted around the parent. So, once you change oPassBulet to use the proper _speed, you need to change their children to call ._speed and ._direction instead of .speed and .direction.
The best way to find these would be doing a codebase-wide search for the name of the object AND the names of any children (i.e. places where they're passed to instance_create.)
This is very fucky for bullets and it may be best to do gDeltaDoTicks for them. This is accomplished by putting

if gDeltaDoTicks != 1 { exit; }

in the first line of the step action of the parent. Check to make sure that the speed isn't "sticky", if it is you may have to set it to 0 on dead frames. If this causes weird effects, flag the object for wareya or brian to rewrite.


2.5) Exception for Effects

An exception to this is visual effect objects which specify speed. These are purely visual and being in the wrong position briefly is fine, so it's OK to ignore them for now, or use the "undo game maker's version of position integration, then do our own" pattern, because it doesn't have the negative effects it has for simulation-affecting objects.

3) FRICTION

Any gameplay-affecting `friction`-having objects (as in, the built in game maker variable called exactly `friction`) should be modified to not use the `friction` variable. Objects doing friction manually with other variables is fine. Summon Brian/Wareya.

5) SUBPIXEL MOVEMENT (Wareya/Brian are probably mostly fixing this)

The base moveTo function is totally fucked and rounds/fractions enemy movement. This will slow them down like up to 20% used natively. (Because like a 1.2 acceleration is neatly 1 + 1/5, but 0.6 acceleration is never 1 + 1/x or something). For now, I'm letting this happen with the hope/expectation that the moveTo function can be replaced with something (like moveToEdit) that does not have this problem. I have some code from Claude that *does* seemingly fix this with high accuracy, but I only used a few times before deciding to put it in the backburner as a high-complexity issue. The Claude code is as follows: 

---Create Event

xRemainder=0
yRemainder=0

---Step event with moveTo

replace the line moveTo(xVel,yVel) with:

xRemainder+=xVel
xMove=round(xRemainder)
xRemainder-=xMove

yRemainder+=yVel
yMove=round(yRemainder)
yRemainder-=yMove

moveTo(xMove,yMove)

6) oGame.time mods

`oGame.time30` contains the number of 30fps ticks that have passed. `oGame.time` contains the number of arbitrarily high-framerate ticks that have passed.

Often, to make something "flip" every xth frame, there will be code like

	if oGame.time mod 6=0

Historically, it seems like this wants to be changed to

	if current_frame mod (6/gDeltaTime)=0
	
oGame.time is seemingly not functional due to our changes or 8.2 or something, so we change that to current_frame, then we divide the intended frame by gDeltaTime to slow it down. Instead of checking every 6th frame it will check every 12th on 60fps and every 24th on 120FPS.

current_frame is 8.2 Core exclusive, so be sure to install the 8.2 Core extension.


7) Velocity Checks (***This may be a bit outdated, consider this only a casual reference; Wareya knows best)

Most of the time, when a number is being checked, we don't want to alter the check. For example, if atkTime is being incremented by 1 a frame and it's being checked for when it hits 10, we want to make it increment 0.5 times a frame and avoid touching the 10. But for some reason, in the following code...

  y+=yVel
  if yVel<3
  yVel+=grav
 
We actually DO want to halve the "yVel" check. It's checking if it's less than 3; we want it to check if it's less than 3/2. Basically, something stuck at 1.5 velocity in 60fps is seemingly going as fast as something going 3 velocity in 30fps. So despite being a check, we do want to modify the velocity being checked for. I think. This is just what has worked so far.

8) "0.5 pixels a frame"

It may be that the subpixels get all janky when trying to move half of a y position. What I found to work instead is to move 1 pixel every other frame like so. ----DELTARUNE GUIDE----

I will copy/paste the guide from Deltarune's 60FPS mod which basically all applies to this project

  - Scale per-frame motion, leave one-time events alone. Movement that runs every frame (x += 4) should be modified by the scale factor. A one-time nudge inside a cutscene (x += 60 when a flag is set), or a single knockback hit, must stay exactly as written. The two look almost identical in code, and confusing them is the most common way to break a scene.
  - Velocity and acceleration scale differently. A speed divides by two. Gravity and friction divide by four, because the engine applies them each frame and then moves the object, which stacks two steps of integration. An acceleration treated like a velocity will look correct and measure wrong, and that gap hides bugs for a long time.
  - The cutscene shortcuts skip the safety net. A handful of commands write a value straight onto an object with no frame-rate awareness at all, used for nudging markers, scheduling a delayed change, or easing a property toward a target. Anything physical that goes through them has to be scaled in the value you pass, because nothing downstream will do it for you.
  - Not everything that repeats is a timer. A counter that wraps around a fixed number to drive an animation phase, or to loop a position around a tower, is describing a shape, not a length of time. Scale how fast you move through it and never the wrap distance itself, or the loop falls apart. Decay is its own case: a value multiplied by 0.95 every frame needs a power, not a division, to keep the same rate.
  - Animations can land a frame short. GameMaker advances sprite frames in a 32-bit-style floating point. An animation tuned to stop exactly on its last frame at 30 FPS can stop one frame early at 60, because the smaller step accumulated over twice as many additions drifts just under the line. The safe fix is to set the final frame outright instead of trusting the count to land.
  - A clean compile is not proof it works. UTMT will accept a call to a function that does not exist, or a built-in variable used as if it were a function, and only fall over at runtime. The only real test is the running game.
